## Lead summary (most consequential first)

- **A confirmed, serious genetic finding is disclosed twice — once well, once badly, in the same
  product** (UX9-01, UX9-02): the `severe` seeded couple has a real double-thalassemia-carrier result.
  The web report correctly softens it to "Needs attention." The **PDF** — generated by the same,
  real `computeGatedComposite` pipeline, not hand-typed seed data — headlines the couple as
  **"Excellent"** / **"very encouraging"** / **"Nothing here should give you pause about building a
  life together"** at a score of **42/100**, then two pages later discloses the actual finding as raw
  status codes: **"Thalassemia: Male status is RED | Female is RED"** with no explanation of what that
  means for their future children. The landing page's own marketing copy explains this exact condition
  in plain, warm language ("if both partners carry it, every pregnancy has a 25% risk") — proving the
  team can write this well; the delivered report simply doesn't.
- **"Prospect" is the primary word for one's fiancé(e)** across headers, upload prompts, chart legends
  and notifications (UX9-03) — the app's own step-level microcopy jokes that it won't do this
  ("we promise not to just call them 'the prospect' the whole time," `trustMessages.js:165`), then does,
  repeatedly, including inside the same report screens that otherwise say "Partner," "the two of you,"
  and show a couple photo emoji.
- **An internal-looking QA/calibration panel, including a button literally labelled "(Demo)," ships
  inside the paid consumer report** (UX9-04), alongside unexplained acronyms and a raw statistical
  weight ("w=0.60") shown to a non-technical audience.
- Two of the same three onboarding questions are hand-authored in two different files and have already
  drifted apart (straight vs. curly apostrophes) — a live, verifiable instance of the exact risk that
  duplication creates (UX9-07).
- Positives: the trust-whisper system (`trustMessages.js`) is a genuinely well-calibrated piece of tone
  design — wit is deliberately dialed down for mental/pathology/genomics content, sincere where it
  matters — and the disclaimer footer, confirm-dialogs, and crash-screen copy are all clear, honest,
  and appropriately reassuring without overclaiming.

### [P0] UX9-01 — PDF's overall status/synthesis text is hardcoded to "Excellent"/reassuring regardless of score, radiology or genetic severity
- Flow / Screen: Downloaded PDF report, page 1 "WHAT THIS MEANS FOR YOU TWO" card and 3-stat row —
  `backend/src/services/pdfReport.service.js:395-520` rendering
  `backend/src/services/compatibility/reportSummary.service.js:349-395, 763-790`
  (`computeGatedComposite` → `mapPresentation`).
- Viewport: n/a (PDF, downloaded from both mobile and desktop "Download PDF Report").
- Status: **Observed** — downloaded the real PDF for the live seeded `severe` match (id
  `33f13c70-c51c-492d-958f-bfa9cd827f4b`, score 42) via the authenticated
  `/api/compatibility/matches/:id/pdf` endpoint and rasterized it:
  `backend/scratch/ux_shots/ux9_severe_report_p1.png` (cover + "Excellent"/42 side-by-side),
  `ux9_severe_report_p4.png` (the actual carrier finding, see UX9-02). Confirmed via direct DB query
  (`backend/scratch/ux9_check_presentation.js`) that `matches.presentation_json.relationship_snapshot`
  = `{"score":42,"status":"Excellent","color":"green","description":"You two are starting from a
  strong, well-matched place. Nothing here should give you pause about building a life together."}`.
  This is **not** hand-seeded fabricated text — `backend/scratch/ux_seed.js:2,8,73` explicitly calls the
  real `reportGen.computeGatedComposite`, the same code path a live match runs, so any real couple with
  this clinical profile (both-carrier thalassemia, Grade II varicocele, PCOS morphology, all correctly
  present in the seed) gets this exact output.
- Evidence: In `reportSummary.service.js:354`, `coupleStatus` is set to `'Excellent'` for **any**
  non-null score, and only escalated away from it if `chronicState`/`mfrState` (two of five domains) is
  `'Specialist conversation'` or `'Plan together'` (`:360-368`). Radiology severity and genetic
  carrier-pair status are **never consulted** when writing `coupleStatus`, `coupleStatusDescription`, or
  `coupleSynthesis` (`:763-777`) — even though those exact two domains are precisely what drove this
  couple's score down to 42 (10% + 10% weight in `reportGeneration.service.js`'s composite). The PDF
  layer has a partial patch for this (`pdfReport.service.js:426-435`: if any `body_health` pathology
  domain is flagged yellow/red, downgrade the *status word* "Excellent"→"Good") but that check doesn't
  cover radiology/genetics either, so for this couple "Excellent" survives untouched — visible verbatim
  in `ux9_severe_report_p1.png`, sitting directly above a "Fertility review recommended · WORTH A LOOK"
  card and a page ahead of the double-carrier disclosure.
  **Cross-surface inconsistency:** the web report gets this right — `story/page.js:224,299-307` derives
  the ring/band purely from the same persisted `relationship_snapshot.score`, so the live app correctly
  shows **"42 PTS" / "WATCH SYNERGY"** for this couple (confirmed live,
  `ux9_severe_story_desktop.png` / `ux9_drive_out.json`'s `severe_story_text_desktop`). The PDF a couple
  downloads, prints, or shares with family is thus **strictly less honest** than the web page they just
  read.
- UX impact: trust + comprehension, in the single highest-stakes surface of the product (the artifact
  meant to be read together, or shown to a doctor/family). A couple with a real hereditary risk and a
  below-average composite score is told in bold serif type that things are "Excellent" and they should
  feel nothing "give them pause" — directly contradicting the number they're looking at on the same page
  and the finding two pages later.
- Heuristic: match between system and real world (the words must agree with the number); error
  prevention; consistency across surfaces (web vs. PDF for the identical match).
- Root cause: copy generation logic (`reportSummary.service.js`) authors `status`/`description`/
  `couple_synthesis` from only 2 of the 5 scored domains; PDF's separate downgrade patch is likewise
  domain-incomplete.
- Best-fit direction: derive `coupleStatus`/`coupleStatusDescription`/`coupleSynthesis` from the same
  **final gated `coupleScore`** the ring already uses (e.g. band directly off the number: <60 → "Action
  Advised"/"Worth a look," not off two hand-picked domain states), so it can never again diverge from
  what the score itself says; extend the PDF's `hasHealthFlag` downgrade to also check radiology
  `risk_flags` and `carrier_pair_risk` severity, or retire it in favor of the single score-driven source
  of truth.
- Effort: M
- Blast radius: `reportSummary.service.js`'s `mapPresentation`/`computeGatedComposite` output feeds the
  PDF, the match-list "Recent Activity" score chip, and (per WS7's ideation doc) a proposed future web
  headline card — a single fix here benefits all three.

### [P1] UX9-02 — A real double-carrier genetic finding is disclosed as raw, unexplained status codes ("Male status is RED | Female is RED") with no plain-language translation or child-risk figure
- Flow / Screen: PDF page 4, "Infectious Disease & Genetic Risk" → "Genetic Carrier-Pair Overlap" card —
  `backend/src/services/pdfReport.service.js` (carrier-pair rendering block feeding from
  `presentation.carrier_pair_risk`).
- Viewport: n/a (PDF).
- Status: Observed — `backend/scratch/ux_shots/ux9_severe_report_p4.png`. Full card text: **"CARRIER
  TRAIT OVERLAP OBSERVED — Thalassemia: Male status is RED | Female is RED. Hb Variant: Male is GRAY |
  Female is GRAY."**, followed only by one generic sentence: "Premarital carrier screening checks if both
  partners carry traits for the same genetic condition, such as thalassemia, which could affect future
  children."
- Evidence: This is the single most clinically serious finding in this couple's entire report (both
  partners confirmed carriers of an autosomal-recessive blood disorder — real reproductive stakes) and
  it is the **only** finding in the whole PDF presented as raw internal status vocabulary ("RED"/"GRAY")
  rather than in the narrative house style used for every other card ("Fertility review recommended,"
  "You'd both feel better with steadier sleep," etc.). There is no plain-language name check ("you are
  both carriers"), no risk figure (the standard, well-known "25% chance per pregnancy" figure that this
  same product's own **marketing copy** states plainly, `frontend/src/constants/landingContent.js:44-46`:
  "Carriers are completely healthy — but if both partners carry it, every pregnancy has a 25% risk"), and
  no next step beyond the one boilerplate sentence. By contrast, the **web** report already does better
  for the identical data: `core-engine/story/page.js:637,646-648` labels the same thread "Future kids'
  genetic baseline" / **"Needs attention"** — still thin, but at least in house style, not raw codes.
- UX impact: comprehension + trust, in a two-people/high-stakes context. A couple reading "Male status is
  RED" with no further explanation may not register that this is a real, actionable finding at all — or
  may panic without the context ("25% per pregnancy, not a certainty, genetic counseling available") that
  would let them act on it constructively. It also reads as an engineering artifact ("status is RED") in
  an otherwise polished, editorial-styled deliverable, undercutting confidence in the whole page.
- Heuristic: recognition over recall / plain language; help users understand real-world consequences of
  a result, not just its internal representation.
- Root cause: copy — the carrier-pair card in the PDF template was written to the raw
  `male_status`/`female_status` enum values instead of a narrative sentence (the presentation JSON
  already carries `carrier_pair_risk.thalassemia.narrative` and `.clinical_footnote` fields, per
  `aiPresentation.service.js`'s schema comments — they appear unused by this specific PDF card).
- Best-fit direction: render the same narrative/clinical-footnote fields the web story tab and the
  earlier PDF pages already use elsewhere, translate RED/YELLOW/GRAY to the same "Needs attention/Worth a
  look/Not fully assessed" vocabulary used throughout the rest of the document, and state the concrete
  child-risk figure and recommended next step (genetic counseling) inline, matching the clarity already
  achieved on the marketing landing page for this exact condition.
- Effort: S–M
- Blast radius: PDF genetic-risk card only; the underlying `carrier_pair_risk` object is otherwise
  already well-utilized (web story tab, dashboard "Needs attention" thread).

- Best-fit direction: introduce one shared "partner label" helper (first name if known, else "your

- Effort: M (touches many call sites, but is a mechanical find/replace once a shared helper exists)
- Blast radius: add-prospect wizard copy, report chart legends (chronic/mfr, both `core-engine/*` and the
  orphaned bare `/chronic`, `/mfr`, `/usg` dev routes which share the same pattern), NotificationBell,
  invite-status strings.

  apostrophes) into one constants module both `login/page.js`'s new-user branch and `onboarding/page.js`
  import, the same way `RELATIONS`/`MARRIAGE_TIMELINES` already are.
- Effort: S
- Blast radius: `login/page.js`, `onboarding/page.js` — copy-only.
